- [ ] LangGraph가 무엇인지 이해
  - [ ] LangChain의 한계: 선형 체인으로 표현하기 어려운 복잡한 흐름
  - [ ] LangGraph = 상태 기계(State Machine) 기반의 그래프형 오케스트레이션 프레임워크

- [ ] API 키 로드
  - [ ] `.env` 파일에서 `GOOGLE_API_KEY` 불러오기
  - [ ] `load_dotenv()` + `os.getenv()` 패턴 복습

## 모듈 01: 첫 번째 그래프

> **학습 목표:** `StateGraph`로 가장 단순한 그래프를 만들고, 노드·엣지·컴파일·실행의 4단계 흐름을 익힌다.

- [ ] LangGraph 핵심 구성 요소 개념 이해
  - [ ] **노드(Node)**: 그래프에서 실행되는 함수 (일 처리 단위)
  - [ ] **엣지(Edge)**: 노드와 노드를 연결하는 흐름 (다음에 뭘 할지)
  - [ ] **상태(State)**: 노드 간에 공유되는 데이터
- [ ] `StateGraph` 로 그래프 정의
  - [ ] `StateGraph(상태 타입)` 으로 그래프 객체 생성
  - [ ] `add_node("노드이름", 함수)` 로 노드 추가
  - [ ] `add_edge("시작노드", "끝노드")` 로 엣지 연결
  - [ ] `set_entry_point("노드이름")` 으로 진입점 지정
  - [ ] `add_edge("노드이름", END)` 로 종료 엣지 추가
- [ ] 그래프 컴파일 및 실행
  - [ ] `.compile()` 로 실행 가능한 그래프 객체 생성
  - [ ] `.invoke(초기_상태)` 로 그래프 실행
  - [ ] 반환값: 최종 상태(딕셔너리)

## 모듈 02: 상태(State) 관리

> **학습 목표:** `TypedDict`로 구조화된 상태를 정의하고, 노드가 상태를 어떻게 읽고 업데이트하는지 이해한다.

- [ ] `TypedDict` 로 상태 스키마 정의
  - [ ] `from typing import TypedDict`
  - [ ] 상태 필드를 타입과 함께 명시적으로 선언하는 이유 (타입 안정성, 가독성)
- [ ] 노드 함수의 입출력 규칙
  - [ ] 노드 함수는 상태 딕셔너리를 인자로 받음: `def my_node(state: MyState)`
  - [ ] 노드 함수는 **변경할 필드만** 담은 딕셔너리를 반환 (전체 상태 반환 불필요)
  - [ ] LangGraph가 반환된 딕셔너리를 기존 상태에 자동 병합(merge)
- [ ] 상태 읽기 및 업데이트 실습
  - [ ] `state["필드명"]` 으로 현재 상태 값 읽기
  - [ ] `return {"필드명": 새로운_값}` 으로 상태 업데이트
- [ ] `Annotated` + `add_messages` 리듀서 개념 맛보기
  - [ ] 기본 동작: 같은 키를 반환하면 덮어쓰기(overwrite)
  - [ ] 리스트를 누적하려면 별도 리듀서 함수 필요

## 모듈 03: 조건부 흐름

> **학습 목표:** 조건부 엣지와 라우터 함수를 사용하여 실행 경로를 동적으로 분기한다.

- [ ] 조건부 엣지(Conditional Edge) 개념
  - [ ] 일반 엣지: 항상 동일한 다음 노드로 이동
  - [ ] 조건부 엣지: 상태에 따라 다른 노드로 분기
- [ ] 라우터 함수(Router Function) 작성
  - [ ] 상태를 입력받아 **다음 노드 이름(문자열)** 을 반환하는 함수
  - [ ] `END` 를 반환하면 그래프 실행 종료
- [ ] `add_conditional_edges()` 로 조건부 엣지 등록
  - [ ] 첫 번째 인자: 분기 출발 노드
  - [ ] 두 번째 인자: 라우터 함수
  - [ ] 세 번째 인자: `{"반환값": "이동할_노드"}` 매핑 딕셔너리
- [ ] 실전: 간단한 분기 그래프 만들기
  - [ ] 상태 값에 따라 두 개의 처리 노드 중 하나를 선택하는 그래프 구현
  - [ ] `from langgraph.graph import END` 임포트 확인

## 모듈 04: 대화형 챗봇

> **학습 목표:** `MessagesState`를 사용하여 대화 기록을 관리하고, 루프 구조로 동작하는 대화형 챗봇 그래프를 만든다.

- [ ] `MessagesState` 이해
  - [ ] `from langgraph.graph import MessagesState`
  - [ ] `messages` 필드를 내장한 미리 정의된 상태 타입
  - [ ] `add_messages` 리듀서: 메시지를 **덮어쓰지 않고 누적** 추가
- [ ] LLM 호출 노드 작성
  - [ ] `state["messages"]` 에서 대화 기록 읽기
  - [ ] `llm.invoke(state["messages"])` 로 LLM 호출
  - [ ] `return {"messages": [ai_response]}` 로 AI 응답 추가
- [ ] 챗봇 그래프 구성
  - [ ] `chatbot` 노드 → `END` 엣지 연결
  - [ ] `.invoke({"messages": [HumanMessage(content="...")]})` 로 실행
  - [ ] 반환된 `messages` 리스트에서 마지막 메시지 확인
- [ ] 멀티턴 대화 구현
  - [ ] 이전 호출의 최종 상태를 다음 호출의 초기 상태로 전달하여 대화 이어가기
  - [ ] 대화 기록이 누적되는 것 확인

## 모듈 05: 도구를 가진 에이전트

> **학습 목표:** `ToolNode`를 그래프에 추가하고, LLM이 스스로 도구 호출 여부를 판단하는 에이전트 루프를 구현한다.

- [ ] `ToolNode` 이해
  - [ ] `from langgraph.prebuilt import ToolNode`
  - [ ] `ToolNode(tools=[...])` 로 도구 목록을 가진 노드 생성
  - [ ] LLM이 `tool_calls` 를 포함한 메시지를 반환하면 자동으로 도구 실행
- [ ] LLM에 도구 바인딩
  - [ ] `llm_with_tools = llm.bind_tools(tools)` 로 도구 정보를 LLM에 전달
  - [ ] LLM이 도구 호출이 필요하다고 판단하면 `AIMessage.tool_calls` 가 채워짐
- [ ] 에이전트 루프 그래프 구성
  - [ ] `agent` 노드: LLM 호출 (도구 사용 여부 판단)
  - [ ] `tools` 노드: `ToolNode` (실제 도구 실행)
  - [ ] `agent` → 조건부 엣지 → `tools` 또는 `END`
  - [ ] `tools` → `agent` (결과를 LLM에게 다시 전달)
- [ ] `tools_condition` 으로 분기 단순화
  - [ ] `from langgraph.prebuilt import tools_condition`
  - [ ] `tool_calls` 유무를 자동으로 판별하는 내장 라우터 함수
  - [ ] `add_conditional_edges("agent", tools_condition)` 패턴
- [ ] 실전: 간단한 계산기 도구를 가진 에이전트 만들기
  - [ ] `@tool` 데코레이터로 도구 정의 (LangChain 모듈 09 복습)
  - [ ] 에이전트 그래프 실행 후 `messages` 기록으로 도구 호출 과정 확인
